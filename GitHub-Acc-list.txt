Clarifying Development and Production Setup...
Short answer: they are two different servers in your setup.

http://localhost:5173/

Frontend dev server (Vite).
Serves the Vue SPA while you develop (HMR, fast rebuilds).
In dev it typically proxies API calls (e.g. /api/*) to the backend so you can call the API without CORS issues.
Use this while actively editing frontend files.
http://localhost:5064/

Backend server (ASP.NET Core / Kestrel).
Hosts the API endpoints (e.g. /api/account/me, /api/admin/users) and in production serves the built SPA from wwwroot.
Also manages authentication, DB access, migrations, etc.
Key practical differences and notes:

Role in workflow:
Develop frontend â†’ open 5173 (fast dev loop).
Test API or run full integrated server â†’ use 5064 (runs backend).
Proxy / cookies:
Vite can proxy /api requests from 5173 to 5064 so the browser treats calls as same-origin. That makes cookie-based auth work during dev if axios.defaults.withCredentials = true and Vite proxy is configured.
If you open 5064 directly (production build served by backend), cookies are naturally same-origin.
Production vs dev:
Production flow: build SPA (npm run build), assets land in wwwroot, then the backend (5064) serves the static files + API.
Dev flow: run Vite (5173) + run backend (5064) simultaneously.
How to tell which one handled a request:
Open browser DevTools â†’ Network â†’ inspect request headers / response. Requests to /assets/* are from Vite (5173) in dev; requests to /api/* go to backend (5064) (or proxied to it).
Or curl each address: curl http://localhost:5173 vs curl http://localhost:5064/api/home.

q6: JWT (JSON Web Token) authentication
---------

JWT (JSON Web Token) authentication is a stateless, token-based method for securely transmitting information between a client and a server, commonly used for web applications and APIs. A JWT contains claims about a user or session, is digitally signed for integrity, and is sent with each request, eliminating the need for the server to store session state.  
How it works
Authentication: A user logs in, and the server authenticates them. 
Token issuance: After successful authentication, the server generates a JWT and sends it back to the client. 
Token structure: The token consists of three parts: a header, a payload (containing the claims), and a signature. 
Header: Contains information about the token type and the algorithm used for the signature. 
Payload: Contains the "claims," which are statements about the user and any other necessary metadata, such as a user ID, name, or permissions. 
Signature: A cryptographic signature is created using the header, payload, and a secret key. This is used to verify the token's authenticity and integrity. 
Subsequent requests: The client sends the JWT to the server with every subsequent request, typically in the Authorization header. 
Verification: The server uses its secret key to verify the signature on the incoming token. If the signature is valid, the server trusts the claims within the token and allows the request to proceed, without needing to look up session data in a database. 
Key benefits
Stateless: The server does not need to store session state, which is ideal for distributed systems and microservices. 
Scalable: Because the server doesn't need to maintain sessions, it can scale more easily. 
Compact: JWTs are a compact way to transmit information between parties, making them efficient to send in HTTP headers. 
Secure: JWTs are signed, ensuring that they have not been tampered with. This can be further secured by using HTTPS to prevent the token from being intercepted. 
Self-contained: All the necessary user information is in the token itself, reducing the need for database lookups on each request.

This prevents the common pitfall where destructuring a reactive object would break reactivity â€” toRefs preserves reactivity for each property when you unpack them.

toRefs is shallow: the ref for list points at the array; nested objects inside the array remain the same references.
Spreading toRefs into the return can accidentally expose more internal state than needed or create naming collisions; prefer returning only the specific refs you need (or use toRef for a single property).

shallow copy:

// array shallow copy
const a = [{ x: 1 }, { y: 2 }];
const b = [...a];
b[0].x = 9;       // a[0].x is also 9 because inner objects are shared

// object shallow copy
const o1 = { a: { n: 1 }, b: 2 };
const o2 = { ...o1 };
o2.a.n = 5;       // o1.a.n is also 5


A shallow copy creates a new container (array or object) whose top-level elements/properties are copied, but if those elements are references (objects, arrays, functions), the references themselves are copied â€” not the nested data. So both original and copy point to the same nested objects; mutating a nested object in one affects the other.

Examples (JavaScript):

Contrast with deep copy: a deep copy recursively clones nested objects so the copy is completely independent. Ways to perform deep copies in JS:

structuredClone(obj) â€” modern, handles many types and preserves structure (recommended where available)
JSON.parse(JSON.stringify(obj)) â€” simple but loses functions, Dates, undefined, Maps, Sets, and fails on circular references
lodash.cloneDeep(obj) â€” reliable library solution
When to use which:

What is SPA?
==================

A SPA (Singleâ€‘Page Application) is a web application that loads a single HTML page and dynamically updates that page in the browser as the user interacts with the app. Instead of requesting full new HTML pages from the server for every navigation, the client-side JavaScript (frameworks like Vue, React, or Angular) swaps views/components and updates the URL via client-side routing to give the feel of multiple pages.

SPAs use client-side routing (e.g., Vue Router) to map URLs to components and often fetch data from APIs (JSON) rather than rendering pages on the server. This enables fast, appâ€‘like navigation, fine-grained UI updates, and rich interactivity because only the necessary data and components are loaded or replaced.

Benefits include smoother navigation, fewer full-page reloads, and better perceived performance after the initial load. Common challenges are larger initial bundle size, SEO and crawlability (addressed with SSR or prerendering), browser history management, and the need for proper caching and codeâ€‘splitting to optimize load times.

To mitigate SPA limitations, teams often use server-side rendering (SSR) or hydration (Nuxt, Next), static site generation (SSG), routeâ€‘level code splitting, and well-configured metadata/SSR for SEO and social previews.

Kebab-case
-----------------------------------------------------------

Kebab-case is a naming convention where words are all lowercase and separated by hyphens (-).
In Vue templates, component tags use kebab-case.

For example, if you register a component as ErrorAlert (PascalCase) in your script, 
you use it in the template as <error-alert></error-alert> (kebab-case).

Why?
Vue automatically converts PascalCase component names to kebab-case when matching tags in the template. 
This is why <error-alert> refers to the ErrorAlert component.
